### 21. 実機フィードバック: 通知アイコンが出ない → 権限フォールバック欠如が原因

- **指摘**: 時計横のステータスアイコンが表示されなかった
- **原因**: MainActivity.onCreateでの通知許可(POST_NOTIFICATIONS, API33+)リクエストが起動時1回のみで、結果を一切確認しないコールバック(`{}`)。位置情報許可と同時に複数ダイアログが並ぶ・見逃す・誤って拒否するといずれの場合も**二度と聞かれず**、Android 13+の仕様上FGSの通知は権限なしでは表示自体されない(サービス自体は裏で動くため他機能は正常に見える紛らわしい失敗モード)
- **修正**: MapScreen側で計測開始(isRunning→true)のたびに`NotificationManagerCompat.areNotificationsEnabled()`を再チェックし、未許可ならAPI33+はその場でリクエスト、拒否済み(再ダイアログが出ない)ならアプリ内ダイアログで「設定を開く」導線(ACTION_APP_NOTIFICATION_SETTINGS)を提示。API33未満(概念上ランタイム許可が無い機種)も同じ導線でカバー
- **教訓**: 「権限リクエストの結果を握りつぶすコールバック」は、サービス自体が動いているせいで不具合に気づきにくい典型パターン。以後、通知系の権限は機能起動のたびに再確認する設計を標準にする

### 22. 通知アイコン不具合(続報): FGS通知の即時表示フラグ漏れ

- **指摘**: 通知許可の再確認フォールバック(21節)を入れても、まだアイコンが出ない
- **調査**: 実機未接続でadb診断できず。コードを再点検し、Android 12+のFGS通知は既定で初回表示が最大10秒遅延・短命サービスとして抑制されることがある仕様と判明。`NotificationCompat.Builder`に`setForegroundServiceBehavior(FOREGROUND_SERVICE_IMMEDIATE)`が未設定だった

- **保留**: これでも直らない場合は実機のUSB接続+adb logcat/dumpsys notificationでの実診断が必要(21節の通知許可自体が実は許可されていない可能性、OEM独自の通知抑制設定の可能性も残る)

### 23. 通知アイコン不具合(根本原因確定): 「サイレント」区分がステータスバーアイコンを隠していた

- **調査**: 実機USB接続でadb診断。権限許可済み・DND OFF・FGS isForeground=true・通知は`stats`に実際のクリック/表示履歴ありと判明 → 通知自体は機能している
- **決定打**: オーナーに通知シェードを引いてもらったところ、通知は「**サイレント**」区分にアイコン付きで表示されていた。Android新UIでは重要度IMPORTANCE_LOW(サイレント扱い)の通知は、フォアグラウンドサービスであってもシェードには出るが**時計横のステータスバーアイコンだけ意図的に省略**される仕様と確定(21-22節の権限・即時表示フラグはどちらも無関係で正常だった)
- **修正**: チャンネル重要度をIMPORTANCE_LOW→IMPORTANCE_DEFAULTに変更。ただし更新頻度が高い(計測のたび)通知が毎回鳴る事態を避けるため、`setSound(null,null)`+`enableVibration(false)`で明示的に無音化(`setOnlyAlertOnce(true)`と合わせ「音は鳴らないがサイレント区分には入らない」状態を実現)
- **重要な副対応**: NotificationChannelは一度作成すると重要度を後から変更できないAndroidの仕様があり、この実機には既に`measurement_status`チャンネルがLOWで作成済みだった。コード側の変更だけでは既存インストール分に反映されないため、**チャンネルIDを`measurement_status_v2`に変更**(新規作成を強制)し、旧チャンネルは`deleteNotificationChannel()`で掃除
- **教訓**: 「通知が出ない」系の不具合は、権限・FGS即時表示フラグだけでなく**通知シェードでの実際の分類(サイレント/通常)**まで実機で確認しないと真因に辿り着けない。IMPORTANCE_LOWは「音を鳴らさない」だけでなく「ステータスバーアイコンも省略されうる」ことを前提に設計する

### 24. 実機フィードバック: ステータスバー5G+なのにアプリはLTE表示 → NSA構成で先頭セル判定が破綻

- **指摘**: 5G(ミリ波)エリア内でステータスバーは「5G+」なのに、アプリの計測表示は「LTE・Band 3」(実機スクショで確認)
- **原因**: `NetworkInfoCollector`が`allCellInfo`の**先頭セルだけ**で種別・バンド・RSSIを判定していた。5G NSA構成(LTEアンカー+NRセカンダリセル)では在圏(isRegistered=true)セルはLTEアンカー側で、NRセルはisRegistered=falseのままリスト後方に報告される端末が多く、先頭セル判定では常にLTE扱いになる(リストの並び順は端末実装依存=まさに「端末アルゴ依存」の壊れ方)。さらに先頭が非在圏の隣接セルだった場合、隣のセルのバンド・RSSIを拾う潜在バグもあった
- **修正**: セル選択を純粋関数`selectMeasurementCellIndex`に切り出し、優先順「①在圏NR(SA 5G) → ②有効な信号強度を持つNR(NSAセカンダリ) → ③在圏セル(LTEアンカー等) → ④先頭フォールバック」で1つのセルを選び、種別・バンド・RSSI・セルIDを**同一セルから**導出するよう`collect()`を一本化。NSA時はNR側のn77/n257・SS-RSRPが記録されるため、`resolveSignalLevel`のMILLIMETER_WAVE(n257)判定にも正しく到達するようになる
- **限界の明記**: ステータスバーの5Gアイコンは`TelephonyDisplayInfo.overrideNetworkType`(OEM/キャリアポリシー)由来で、購読にはREAD_PHONE_STATEが必要(本アプリはAPI29+で非保持)。allCellInfoにNRセルが現れない端末・タイミング(アイコンが「5G利用可能」を先行表示する場合等)では引き続きLTE記録になりうる。API26-28のdataNetworkTypeパスもNSAをLTEと報告する制約あり(コメントに明記)
- **検証**: 環境制約(Android SDK取得先dl.google.comがネットワークポリシーで遮断)によりgradleビルド不可のため、(1)純粋関数+実テストファイルをkotlin-compiler-embeddable単体でコンパイルし新規テスト8件全パス、(2)実ファイル全体をRobolectricのandroid-all(API35実フレームワーク)クラス群に対してコンパイルしエラーゼロを確認。実機での5Gエリア再訪確認はオーナーに依頼

### 25. 5G/LTE誤判定の続報: NRセルを報告しない端末向けにTelephonyDisplayInfo補正を追加

- **指摘**: 24節の修正後も実機で「ステータスバー5G+/アプリLTE・Band 3」のスクショ報告(※修正版APKの導入有無は未確認。旧ビルドの可能性もあるが、下記の通り旧ロジックでは直らない端末が存在するため先回りで対処)
- **原因(仮説)**: 5G NSAでは端末によって`allCellInfo`にNRセカンダリセルが**一切現れない**実装があり、その場合24節のセル走査(全セルからNR検出)でも在圏LTEアンカーしか見えず、原理的にLTE判定になる
- **調査**: ステータスバーの5Gアイコンと同一情報源`TelephonyDisplayInfo.overrideNetworkType`の購読権限を、Android 15実フレームワーク(Robolectric android-all)のバイトコードとcompat設定で確認。`TelephonyRegistry_compat_config.xml`に「DisplayInfoListenerはtargetSdk 31+でREAD_PHONE_STATE要求を撤廃(REQUIRE_READ_PHONE_STATE_PERMISSION_FOR_DISPLAY_INFO)」と明記されており、本アプリ(targetSdk 35)は**権限追加なしで購読可能**と確定
- **修正**:
  - `DisplayInfoMonitor`(プロセス寿命シングルトン)を新設し、API 31+で`TelephonyCallback.DisplayInfoListener`を購読。`overrideNetworkType`がNR_NSA/NR_ADVANCEDなら「NSA 5G接続中」と判定
  - `NetworkInfoCollector`のLTE判定時に純粋関数`applyNrOverride`で補正: NR接続中ならnetworkType="5G"に昇格。**バンドはnullに落とす**(LTEアンカーの"Band 28"を5G n28プラチナと誤判定するのを防ぐ。RSSI・セルIDは実測可能なアンカーの値を記録)
  - API 30以前はDisplayInfo購読に権限が必要なため購読せず、24節のセル走査のみで判定(制約をコメントに明記)
- **検証**: 24節と同じ環境制約のため、(1)純粋関数テスト13件(セル選択8+NR補正5)をkotlin-compiler-embeddableで全パス、(2)`NetworkInfoCollector.kt`+`DisplayInfoMonitor.kt`をandroid-all(API 35)に対してフルコンパイルしエラーゼロ。**実機確認には修正版APKのビルド・インストールが必要**(オーナー依頼)
